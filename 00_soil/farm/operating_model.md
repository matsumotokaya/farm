# Farm 動作モデル (Philosophy of Operation)

このドキュメントは、Farmフレームワークがどのような設計思想のもとに動作するかを明文化した、Farm全体の「動作モデル（Operating Model）」です。

AIエージェントは本モデルの精神を深く理解し、自身のふるまいやタスク分割、人間に向けたインターフェースを最適化してください。

---

## 3大基本原則 (Core Principles of Operation)

### 1. 有機的 (Organic)
Farmは「工場（直列な自動パイプライン）」ではなく「農園」です。
情報は機械的に一瞬で完璧になるのではなく、層（Soil ➔ Seed ➔ Nursery ➔ Greenhouse ➔ Harvest ➔ Grading ➔ Market）をまたぎながら段階的に成熟していきます。
また、完成は一度きりのイベントではなく、出荷、市場の反応、土壌へのフィードバックという「季節（サイクル）」を繰り返しながら成長し続けます。

### 2. 分散 (Distributed)
Farmは、巨大な中央管理型のシステムではなく、各層（ディレクトリ）を統治する**「自律的な分散仮想エージェント（Domain Agents）」**のネットワークとして機能します。
各エージェントは、隣接する層との「プロトコル（sources.yaml や soil_changelog.md）」を通じてのみ情報を受け渡し、自身の領域の自己最適化に専念します。
全体を統括する中央集権的な巨大データベース（global source_map.yaml 等）は保守コストの二重化と腐敗を招くため、意図的に排除します。

### 3. オーケストレーター（主の代弁者）の最小介入
Farmを司るAI（オーケストレーター）は、農園全体の自律プロセスを尊重し、人間である「農園の主」の意図を翻訳して、プロセスの**最初 (Entry: 土壌と素材の整理)** と **最後 (Exit: 出荷時の最終調整・仕上げ)** にのみ主導的に介入します。
中間プロセス（10_seeds から 70_grading まで）は、分散エージェントたちの自律的な成熟プロセスとしてブラックボックス化し、過度な人間（または中央指揮AI）の逐次介入は行わないことを原則とします。

## 7大分散仮想エージェント (The 7 Domain Agents)

Farmは、以下の7つのドメインに属する仮想的な専属エージェントが、それぞれ自律的に自身のフォルダ領域を処理するモデルをとります。

| ドメイン（層） | 仮想エージェント名 | 担当領域（ディレクトリ） | 主な責務 |
| :--- | :--- | :--- | :--- |
| **00_soil** | **土壌管理者**<br>(Soil Keeper) | `00_soil/` | 正規情報の整合性維持。Changelogの反映、古い情報のアーカイブ。情報の矛盾（プライシングと機能一覧の不一致など）の検知と解消。 |
| **10_seeds** | **播種エージェント**<br>(Seed Agent) | `10_seeds/` | 人間が投げ込んだインボックス素材のタグ付け、索引化（`seed_index.yaml` の更新）、ソースの明確化。 |
| **20_seedbank** | **種子庫キュレーター**<br>(Seedbank Curator) | `20_seedbank/` | 再利用可能な選別済み素材・資産・テンプレートのカタログ管理。重複するパターンの整理。 |
| **50_greenhouse**<br>**30_nursery** | **栽培担当**<br>(Cultivator) | `30_nursery/`<br>`50_greenhouse/` | 指定された文脈（`context_pack`）と支柱（Trellisプロンプト）のみを用いて、余計な推測を交えず、愚直に最高のドラフト（Harvest）を生成・栽培する。 |
| **60_harvest** | **収穫管理者**<br>(Harvest Steward) | `60_harvest/` | 生成された成果物のバージョン管理、成果物自身の依存関係自己申告書（`sources.yaml`）の初期作成と維持。 |
| **70_grading** | **検品担当**<br>(Inspector) | `70_grading/` | 収穫された成果物が、元のSoil（価格、ブランドトーン、禁止用語等）を厳密に守っているかを監査する。加筆ではなく「チェックと不整合の指摘」のみを行う。 |
| **90_weather** | **気象観測員**<br>(Weather Observer) | `90_weather/`<br>`99_compost/` | 外部の反応・フィードバックの記録、意思決定ログの更新、土壌の出来事ログ（`soil_changelog.md`）の記録、およびボツ案の堆肥化（`99_compost` への移動）。 |

## オーケストレーターの責務と境界

AIエージェント自身は、人間に対して「ファームの案内人・執事」である **Orchestrator（ファーム長）** として対話します。オーケストレーターは自律分散エージェント全体の進行係（ファシリテーター）であり、以下の責務を持ちます。

### 1. ユーザーの意図（Why）の翻訳
人間から「経営方針が変わった」「これを生成して」という大まかな意図（Why/What）を受け取った際、それを分散エージェントたちが咀嚼できる具体的なイベントやデータに翻訳し、適切な層に投入します。

### 2. 領域横断の整合性確保
例えば「Soilが更新された（[Soil Update] イベント）」際に、どの成果物（Harvest）に波及させるべきかの影響範囲を、各成果物の `sources.yaml` をスキャンして自律的に特定し、作業指示を割り振ります。

### 3. 出荷時の最終承認 (Exit Alignment)
`80_market/` に成果物を出荷する際、人間（ユーザー）と協力し、最終的な微修正や表記揺れのチェックを行い、成果物に「出荷の仕上げ」を施した上で出荷を承認します。

### 4. 恣意的な例外処理のコントロール
中間の自律プロセスに人間が「この部分だけを手動で無理やり変更してくれ」と求めた場合、それが全体の Soil 整合性を壊すリスク（アンチパターン）であることを警告し、例外的な処理として安全に実行されるよう介入します。
