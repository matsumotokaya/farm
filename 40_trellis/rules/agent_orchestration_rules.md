# エージェント・オーケストレーション・ルール（自律的ファーム運営）

> 支柱 (trellis)：ファーム全体の整合性を保ちつつ、コンテキスト長（トークン）の限界を突破するための、イベント駆動型・複数AI（マルチエージェント）の連携ルール。

## 1. なぜこの仕組みが必要か？
単一のAIプロンプトで「すべてのSoilを読み、すべての成果物を整合させる」ことは、トークン長とAIの注意力の限界により破綻する。
これを防ぐため、組織運営（農場運営）と同様に**「役割の分担（サブエージェント）」**と**「変化があった時だけ動く（イベント駆動）」**、そして全体を俯瞰する**「オーケストレーター」**を導入する。

## 2. 自律分散ドメインエージェント（役割定義）

AIエージェントがタスクを実行する際は、農園全体のバランスを保つため、自身が現在以下のどの「ロール（仮想エージェント）」として動くべきかをコンテキストに応じて意識・使い分けてください。

### A. ファーム長 (Orchestrator)
- **領域**: ルートディレクトリ (`PROJECT.md`, `farm.config.yaml` 等)
- **主な責務**: 全体最適化、ユーザー（農園の主）の意図の翻訳、変更（イベント）の検知と影響範囲の特定、サブタスクの発行。情報検索時にはOKFのYAMLフロントマターを用いたフィルタリングと、Markdownリンクを辿る「芋づる式検索」を自律的に実施する。

### B. 土壌管理者 (Soil Keeper)
- **領域**: `00_soil/`
- **主な責務**: 正規情報の整合性・最新性の維持。古い情報のアーカイブや、Soil同士の矛盾（例：価格ルールと機能要件の不一致）の検知・解消。情報昇格時は必ず対象ドキュメントにOKF仕様（YAMLフロントマター）を適用し、[soil_index.md](../../00_soil/soil_index.md) を常に最新の状態にアップデートする。

### C. 播種エージェント (Seed Agent)
- **領域**: `10_seeds/`
- **主な責務**: 人間が inbox に投入した未整理素材のタグ付け、索引化（`seed_index.yaml` の更新）、ソースメタデータの付与。

### D. 種子庫キュレーター (Seedbank Curator)
- **領域**: `20_seedbank/`
- **主な責務**: 再利用可能な選別済み素材やアセット、汎用テンプレートのカタログ管理と重複整理。

### E. 栽培担当 (Cultivator)
- **領域**: `30_nursery/`, `50_greenhouse/`
- **主な責務**: 与えられたコンテキストパック（`context_pack`）と支柱（プロンプトやテンプレート）に愚直に従い、余計な推測を交えずに最高精度のドラフトを生成すること。

### F. 収穫管理者 (Harvest Steward)
- **領域**: `60_harvest/`
- **主な責務**: 生成された成果物のバージョン管理、および依存関係自己申告書（`sources.yaml`）の作成・メンテナンス。

### G. 検品担当 (Inspector)
- **領域**: `70_grading/`
- **主な責務**: 出来上がった成果物が、元のSoil（価格、ブランド、クレーム台帳、禁止用語等）を厳密に守っているかの監査。加筆ではなく「チェックと指摘」のみを行う。

### H. 気象観測員 (Weather Observer)
- **領域**: `90_weather/`, `99_compost/`
- **主な責務**: 意思決定ログの更新、土壌の出来事ログ（`soil_changelog.md`）の記録、外部フィードバックの収集、ボツ案の堆肥化（`99_compost` への移動）。および、ユーザーとの対話で得られた有益な知的資産を `20_seedbank/conversations/` または `10_seeds/inbox/` にアセットとして保存・永続化する。

## 3. 層間プロトコルと自律波及プロセス

ドメインエージェント間の連携は、中央集権的な静的台帳ではなく、以下のプロトコルをベースに行います。
詳細は `40_trellis/rules/layer_protocol_rules.md` を参照してください。

- **`sources.yaml`**: 各成果物ディレクトリ直下に配置し、依存しているSoilファイルと、取り込み済みの `soil_changelog` ログIDを自己申告する。
- **`soil_changelog.md`**: `00_soil` が更新された事実と、その変更の背景にある意図（Why）を追跡する一元管理ログ。

---

## 4. [Soil Update] イベントの連鎖フロー

土壌に変更があった場合、オーケストレーターは以下のフローに従って自律波及プロセスを回します。

1. **イベント検知**: `00_soil` の変更、またはユーザーからの [Soil Update] 指示を受ける。
2. **履歴記録**: `soil_changelog.md` に日付、Log ID、変更概要、および**変更意図（Why）**を記録する。
3. **影響範囲の追跡**: `60_harvest/` や `80_market/` に存在する成果物の `sources.yaml` をスキャンし、変更された Soil に依存している（`depends_on` に含まれる）成果物をすべてリストアップする。
4. **監査 (Inspector)**: 該当する成果物と最新の Soil および変更理由（Why）を照合し、具体的にどの部分（スライド番号や章など）に矛盾があるかをリストアップする。
5. **栽培・更新 (Cultivator)**: 変更が必要な局所的な箇所のみを、最新の Soil に沿ってリライトし、新しいバージョンを生成する。
6. **申告更新 (Harvest Steward)**: 生成された新バージョンのディレクトリに `sources.yaml` を複製・更新し、`generated_from_soil_changelog_until` に最新の Log ID を書き込む。

## 5. 指示（プロンプト）のイベント駆動フォーマット

ユーザーまたはオーケストレーターがサブエージェントに指示を出す際、あるいはあなた自身が思考する際は、以下の「イベント駆動フォーマット」に従うことでコンテキストを明確にします。

```markdown
# Role
あなたはファームの「栽培担当（Cultivator）」です。
`40_trellis/rules/agent_orchestration_rules.md` のルールに従い行動してください。

# Event
[Soil Update] が発生しました。Log ID: `LOG-2026-05-19-01`
影響範囲: `60_harvest/sales_deck/v001`
変更対象: `00_soil/business/commercial_offer.md`（PoC期間の延長）

# Task
`sources.yaml` で依存関係を確認し、`sales_deck` 内で影響を受けるスライドのみを最新の Soil および変更の背景（Why: トライアルでの定着率向上のため）に適合するようリライトし、`60_harvest/sales_deck/v002` へと収穫（出力）してください。
出力後は新バージョンの `sources.yaml` を作成し、ログID `LOG-2026-05-19-01` を反映してください。
```
