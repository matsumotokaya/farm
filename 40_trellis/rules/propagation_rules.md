# 情報伝播ルール（Propagation Rules）

> 「土壌（経営方針やコアメッセージ）の変更を、瞬時にすべての枝葉（成果物）へゼロフリクションで反映させる」ためのルールです。

## 基本方針

AI時代の最大の恩恵である「情報伝播のフリクション・ゼロ」を実現するため、本Farmでは、コアとなる `00_soil/` の情報が変更された際、それが末端のあらゆる成果物に自律的かつ一貫して波及する生態系を維持します。
事業方針やプライシングの変更が、即座に営業資料、LP、1枚の広告バナーの文言（`slide_markdown` 等）にまで連動して反映される状態を目指します。

## 自律分散レイヤープロトコルによる波及

従来の「中央台帳（`source_map.yaml`）」による一元管理は廃止し、**成果物ごとの自己申告（`sources.yaml`）**と**変更ログ（`soil_changelog.md`）**を用いた自律波及プロセスを採用します。

1. **変更の検知と依存関係の追跡**
   - `00_soil/` 内の正規情報に変更が加えられた際、オーケストレーターは `90_weather/soil_changelog.md` に日付、Log ID、変更概要、意図（Why）を記録します。
   - `60_harvest/` 配下にある各成果物の直近の有効バージョンの `sources.yaml` をスキャンし、`depends_on` に該当ファイルを含み、かつ `generated_from_soil_changelog_until` が現在の Log ID より古いものを特定します。

2. **カスケード更新の自律的な実行・提案**
   - オーケストレーターは影響範囲を特定後、検品担当（Inspector）に監査タスク、栽培担当（Cultivator）に部分リライトタスクを発行します。
   - 人間が修正箇所を手動で探す必要はありません。システムが自律的に「どこをどう直すべきか」を特定し、新しい Soil と変更の背景（Why）に沿ってアップデートします。

3. **例外的な selective update（部分反映）の制御**
   - 特定の成果物で古い設定を維持したい場合、その成果物の `sources.yaml` の `not_yet_reflected` セクションに未反映の Log ID と理由を明記することで、一括伝播の対象外として例外管理します。
