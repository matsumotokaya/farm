# Farm Rules

> 唯一のルールファイル。v1では12ファイル1,500行あったルールを、機構（farm.sh）に置き換えた上で、機構にできない規律だけをここに残した。
> ここに新しいルールを足したくなったら、まず「それはスクリプトか構造変更で保証できないか」を検討する（README設計原則1）。

## 1. soilの規律（正史の育て方）

- **サイズ上限**：10ファイル目安、1ファイル300行目安。「生成前に全部読める」ことがこの上限の理由であり、超えそうなときはファイルの統合か`deprecated`化を先に行う
- **frontmatter必須**：`title` / `summary` / `status` / `updated` / `tags`。`barn/templates/soil.md` を使う
- **statusの意味**：
  - `canonical` — 生成に使ってよい唯一の状態
  - `draft` — soil内で検討中。生成には使わない
  - `deprecated` — 歴史として残すが生成には使わない。3か月動きがなければ `9_compost/` へ移す
- **正の定義で書く**：soilには「何であるか」を書く。「何でないか」「何に寄りすぎないか」のような矯正メモは、成果物本文に漏れる事故の源になるため、書く場合は各ファイル末尾の`## 書き方の注意`節に隔離し、本文とは分ける
- **主張と数字は claims.md に集約する**：外部に出す主張・数字は `0_soil/claims.md` に登録し、成熟度を持たせる
  - `validated` — 出典確認済み。外部提出物の本文に使える
  - `pending` — 出典待ち。内部ドラフトのみ。外部提出前にvalidatedへ昇格させるか脚注へ落とす
  - `desired` — あれば説得力が上がる数字。調査要求として扱う（「足りないからできない」ではなく「取りに行くリスト」）

## 2. seedsの規律（受信箱）

- **追記のみ**。編集・分類・索引化の義務はない。ファイル名 `YYYY-MM-DD_<topic>.md` だけ守る
- seedsからsoilへの昇格は蒸留の儀式で行う。**昇格は1段だけ**（中間の選別庫や苗床は作らない）
- 昇格時、元seedは消さずに残す（出自の追跡はファイル名の日付とgit履歴で足りる）

## 3. harvestの規律（成果物のコンパイル）

- 成果物ごとに1ディレクトリ（例：`2_harvest/business_plan/business_plan.md`）
- **版番号ディレクトリを作らない**。過去版はgit履歴とタグで参照する
- frontmatterマニフェスト必須：`artifact` / `audience` / `status` / `depends_on` / `built_from` / `built_at`（`barn/templates/artifact.md`）
- **depends_onは正直に**：生成に実際に読んだsoilファイルを列挙する。これがstale検出の根拠になる
- **再生成が原則**：依存先soilが変わったら、該当セクションの手直しではなくsoilからの再生成で追従する。soilが正しければ再生成は安全である（逆に、再生成が怖いと感じたら、それはsoilに書かれていない決定が成果物に直書きされているサイン。その決定をsoilへ昇格させる）
- 成果物の中で新しい主張・数字・言い回しを発明しない。必要なら先にsoil（claims.md / terms.md）へ足してから使う

## 4. 二つの帽子（生成と監査の分離）

- **生成の帽子**：soilを読み、最良のドラフトを作る。批評はしない
- **監査の帽子**：加筆せず、チェックだけを行う。soilとの整合、claims.mdとの数字一致、表現、読者視点の穴
- 同一コンテキストで両方をやると監査が甘くなる。出荷前の監査は別セッションで行うか、「ここから監査の帽子で見ます」と明示的に宣言して行う

## 5. 出荷（ship）

1. 監査の帽子でチェックを通す
2. 成果物frontmatterの `status` を `shipped` にし、`built_from` を最新commitに更新する
3. `git tag ship/<artifact>/<YYYY-MM-DD>` を打つ。誰に何を出したかはタグメッセージに書く
4. 出荷後の外部反応は `1_seeds/feedback/` に入れる（次の季節の種になる）

## 6. 表現統制（v1で確立された最重要の運用知見）

表現の制御は「使わないでという指示」ではなく「物理的な削除・置換」で行う。

1. 対象表現をgrepで全ファイルから検索する
2. すべて置換・削除する
3. grepで残存ゼロを確認する

理由：文書内に特定の単語が残っている限り、AIはそれを使えると判断する。伝播を止める唯一の方法は物理的に消すことである。過去ログで旧表現に言及するときは「旧表現」のように抽象化し、具体単語を残さない。恒久的な対象単語リストは持たない（リスト自体が単語の温床になる）。

## 7. 検索の仕組み（RAG）

Farmの検索は3層で、すべて機構が支える。

1. **索引** — `0_soil/_index.md`（farm.sh indexが自動生成）。AIはまずここを読み、どのsoilを開くか決める
2. **frontmatterフィルタ** — `status: canonical` だけを真実として扱い、`tags` で絞り込む
3. **リンク追跡（Traversal）** — soilファイル末尾の `## 関連` のMarkdownリンクを辿り、周辺知識を芋づる式に収集する。soilを書くときは関連リンクを1〜3本置く

## 8. Farm Coreへの還流

Instance運用中に「これはCoreのデフォルトにあるべきだった」と気づいたら、`FARM_CORE_FEEDBACK.md` に追記する（気づき・提案・優先度の3点）。送るかどうかは人間が判断する。
